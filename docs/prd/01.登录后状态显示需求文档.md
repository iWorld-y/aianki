# 登录后状态显示需求文档

## 1. 概述

### 1.1 背景

登录功能已开发完成，需要明确登录后各页面的数据展示策略，包括：

- 数据加载时机和刷新机制
- 各页面展示的具体数据内容
- 空状态、加载状态和错误状态的处理

### 1.2 目标

- 提供流畅的用户体验，数据加载无感知
- 保证数据一致性，多页面间状态同步
- 优雅处理网络异常和数据为空的情况

| `stats.streakDays`    | API: `GET /stats` | 连续学习天数 |
| `stats.masteredCards` | API: `GET /stats` | 已掌握卡片数 |

#### 3.3.2 统计数据定义

```typescript
interface UserStats {
  totalCards: number; // 用户创建的所有卡片总数
  todayReviewed: number; // 今日已完成复习的卡片数（不设置上限）
  streakDays: number; // 连续复习天数（按自然日计算：今天或昨天有复习则延续）
  masteredCards: number; // 状态为"review"且稳定性>7的卡片
}
```

**统计计算规则**：

- **今日已复习**: 自然日 00:00-23:59 内完成复习的卡片数，不设置上限
- **连续天数**: 按自然日计算，今天或昨天有复习记录则连续天数+1，断开后重置为0
- **已掌握**: 卡片状态为"review"且稳定性(stability)大于7天

| 无连续学习 | 连续天数显示"0"，提示"坚持学习建立习惯" |

## 4. API 接口规范

### 4.1 接口列表

| 接口       | 方法 | 路径                         | 说明               |
| ---------- | ---- | ---------------------------- | ------------------ |
| 今日待复习 | GET  | `/review/today`              | 获取今日待复习卡片 |
| 用户卡组   | GET  | `/decks`                     | 获取用户所有卡组   |
| 最近卡组   | GET  | `/decks?limit=5&sort=recent` | 获取最近使用的卡组 |
| 统计数据   | GET  | `/stats`                     | 获取用户统计数据   |
| 创建卡组   | POST | `/decks`                     | 创建新卡组         |
| 提交复习   | POST | `/review/submit`             | 提交复习结果       |

### 4.2 响应格式

```typescript
// 统一响应格式
interface ApiResponse<T> {
  code: number; // 0 表示成功，其他表示错误
  message?: string; // 错误信息
  data: T; // 业务数据
}
```

### 4.3 错误码

| Code | 含义       | 处理建议                   |
| ---- | ---------- | -------------------------- |
| 0    | 成功       | -                          |
| 401  | 未授权     | 清除token，跳转登录        |
| 403  | 权限不足   | 提示"无权限"               |
| 404  | 资源不存在 | 提示"数据不存在"           |
| 500  | 服务器错误 | 提示"服务繁忙，请稍后重试" |
| 1001 | 参数错误   | 提示具体参数问题           |

## 5. 数据加载流程

### 5.1 页面加载时序图

```
Page.onLoad/onShow
    │
    ▼
检查登录状态 (app.globalData.isLoggedIn)
    │
    ├── 未登录 ──► 显示游客模式
    │
    └── 已登录 ──► 检查缓存有效性
                      │
                      ├── 缓存有效 ──► 使用缓存数据
                      │
                      └── 缓存失效 ──► 调用 API
                                          │
                                          ├── 成功 ──► 更新数据+缓存

                                          └── 失败 ──► 显示错误/重试按钮
```

## 6. 数据同步策略

### 6.1 多端同步

- **触发时机**: 每次操作后（创建、复习、删除）
- **同步方式**: 主动推送 + 定时拉取
- **冲突处理**: 以服务器时间为准

### 6.2 统计数据实时计算

- **计算方式**: 统计数据（`totalCards`、`todayReviewed`、`streakDays`）每次请求时实时计算
- **触发时机**:
  - 页面加载/刷新时
  - 完成复习提交后（自动刷新统计数据）
  - 创建/删除卡片后
- **优化策略**: 对于高频操作，可先在本地预计算，API 返回后再校正

### 6.3 离线支持

- 暂不支持离线创建和复习
- 网络异常时提示用户检查网络
- 关键操作（复习提交）需要网络确认

### 6.4 数据保留策略

- **退出登录**: 不清除本地缓存数据（`userInfo`、`decks`、`stats`等），下次登录同账号可快速恢复
- **账号切换**: 新账号登录时自动覆盖旧数据
- **缓存清理**: 用户手动清理小程序缓存时一并清除

## 7. 性能优化

### 7.1 加载优化

- **骨架屏**: 页面加载时显示占位内容
- **分页加载**: 卡组列表使用无限滚动
- **图片懒加载**: 卡片图片延迟加载
- **数据预加载**: 预测用户行为提前加载

### 7.2 缓存策略

- **内存缓存**: 页面级别数据缓存
- **本地存储**: 用户信息和统计数据
- **CDN 缓存**: 静态资源（头像、图标）

## 8. 异常处理

### 8.1 网络异常

| 场景         | 处理策略               |
| ------------ | ---------------------- |
| 首次加载失败 | 显示重试按钮           |
| 下拉刷新失败 | Toast 提示，保留旧数据 |
| 提交操作失败 | 保留表单数据，提示重试 |

### 8.2 数据异常

| 场景         | 处理策略                   |
| ------------ | -------------------------- |
| 数据格式错误 | 记录日志，显示默认数据     |
| 数据缺失字段 | 使用默认值填充             |
| 数据过大     | 分页加载，限制单次返回数量 |
